# HedHog Lab v2

These instructions apply to the entire repository.

## Global Rules

- Respect monorepo architecture: business logic in `libraries/*`, app entrypoints in `apps/*`, shared utilities in `packages/*`.
- Use workspace package imports (`@hed-hog/*`) and avoid deep relative imports across workspaces.
- Keep changes minimal and consistent with existing module/app style.
- Read only the files needed for the current task; do not scan the whole repository without a concrete reason.
- For large, risky, or multi-domain changes, state the intended approach before editing.
- For admin CRUD/list pages, follow this canonical layout: `PageHeader` -> KPI row -> flat `SearchBar`/filters + `Tabela`/`Cards` toolbar -> content -> `PaginationFooter`, without extra wrapper cards or duplicate headings under `PageHeader` unless product context requires it. Once a library page exists under `apps/admin/src/app/(app)/(libraries)/<library>/<entity>/page.tsx`, treat it as the concrete reference (this folder is empty in a fresh template checkout — it is generated by `hedhog dev create-library` / `assets-to-library`).
- For frontend interactions, clickable rows/cards/list items must expose clear affordance using `cursor-pointer` (or an equivalent component style that renders a pointer cursor).
- For admin searchable relation/entity selectors with inline create, prefer the shared `EntityPicker` at `apps/admin/src/components/ui/entity-picker.tsx` (import path `@/components/ui/entity-picker`) instead of creating new ad-hoc `*FieldWithCreate` or `*SelectWithCreate` implementations. Thin semantic adapters are acceptable only when they delegate to `EntityPicker`.
- For admin forms that require draft persistence, use the shared hook at `apps/admin/src/hooks/use-form-draft.ts` instead of custom localStorage logic. Drafts must remain isolated per user, recoverable for the same user after involuntary auth interruption, cleared on successful submit, and purged on manual logout; never persist passwords, tokens, or raw file/blob payloads.
- Shared components first (mandatory): for recurring UI flows (create/edit sheets, dialogs, entity pickers, and complex form sections), reuse or extend shared components in `apps/admin/src/components/**` and avoid duplicating full flow logic in page files.
- If a page-local implementation is introduced as a temporary step, extract it to a shared component before task completion.
- For any new Sheet in `apps/admin`, use `ResizableSheetContent` (instead of raw `SheetContent`) with a stable `sheetId`, so desktop resize and per-user width persistence are consistent by default.
- New Sheets must preserve mobile full-width behavior through the shared resizable sheet wrapper (`100%`/`100vw` behavior).
- Before finishing, run focused build/test/lint checks in touched packages/apps when feasible.

## Codex Subagents

- `AGENTS.md` is the primary orchestration layer for Codex. Keep it short and use `.codex/subagents/*.md` for role-specific execution prompts.
- `.github/agents/*.agent.md` remain compatibility/reference prompts for Copilot and detailed fallback context; do not load them by default when the Codex subagent file is sufficient.
- Use subagents only when the task is non-trivial and clearly maps to a single owner, or when a mandatory handoff is needed.
- Keep work local for one-file fixes, simple mechanical edits, read-only audits, local analysis, and changes already covered by these rules plus nearby code patterns.
- If a task spans multiple domains, assign the primary owner first, then execute the required handoff chain below instead of loading every specialist up front.

### Dispatch Rules

- If a task is dashboard-focused by intent and non-trivial (even when it touches non-dashboard paths like `apps/api/src`), use `Dashboard` as primary owner and let it coordinate required handoffs.
- Use `Dashboard` for dashboard-scoped work in `apps/admin/src/app/(app)/(libraries)/*/dashboard/**`, `libraries/*/src/dashboard/**`, and `libraries/*/hedhog/frontend/widgets/**`, including widget lifecycle, component catalog, layout/position behavior, dashboard UX, and dashboard backend integration.
- Use `Backend` for non-trivial NestJS runtime changes in `libraries/*/src` or `apps/api/src`, including controllers, services, DTOs, auth, pagination, and Prisma-backed behavior.
- Use `Hedhog YAML` when actually writing `libraries/*/hedhog/table/*.yaml` or non-permission `libraries/*/hedhog/data/*.yaml`; do not invoke it for read-only YAML inspection.
- Use `Route Permissions` when actually writing `libraries/*/hedhog/data/route.yaml` or `libraries/*/hedhog/data/role.yaml`, or when endpoint changes need permission synchronization.
- Use `Hedhog Library` for new library design/bootstrap, cross-library reuse decisions, dependency planning, and table ordering.
- Use `Frontend` for non-trivial changes in `apps/admin`, frontend i18n/messages, forms, lists, and data fetching integration.
- Use `Library Documentation` for creating or refreshing `libraries/*/README.md` from real controllers, DTOs, services, and table YAML.

### Mandatory Handoffs

- `Dashboard` -> `Route Permissions` whenever dashboard endpoint surface changes, including URL, HTTP method, auth/role requirements, or endpoint removal.
- `Dashboard` -> `Hedhog YAML` for writes in `libraries/*/hedhog/table/*.yaml` or `libraries/*/hedhog/data/*.yaml` (non-permission), then continue with runtime integration.
- For dashboard widget catalog/role seeds (`dashboard_component.yaml` and `dashboard_component_role.yaml`), `Dashboard`/`Hedhog YAML` must write in the target library (`libraries/<library>/hedhog/data`), set `library_slug` to the owning library, and must not implicitly edit core equivalents unless core is explicitly requested.
- When dashboard widgets are copied into admin runtime assets, treat the generated file name as `<library>.<widget>` and resolve imports using `library_slug + '.' + slug`.
- `Backend` -> `Route Permissions` whenever endpoint surface changes, including URL, HTTP method, auth/role requirements, or endpoint removal.
- `Hedhog Library` -> `Hedhog YAML` for any write to `hedhog/table/*.yaml` or `hedhog/data/*.yaml`.
- `Frontend` -> run `hedhog dev assets-to-library <libraries...>` before finishing when changes affect `apps/admin/src/app/(app)/(libraries)` or `apps/admin/messages`.
- `Library Documentation` may read code and YAML but must not change runtime code, route seeds, or Prisma schema.

### Token Efficiency Rules

- Prefer the shortest subagent prompt that fully owns the task.
- Do not preload multiple subagent prompts when ownership is clear.
- Reuse the global rules in this file instead of repeating them in every subagent prompt.
- Prefer direct main-agent execution for small and medium-low-risk edits when no specialist context or handoff is needed.
- Read `.github/agents/*.agent.md` only when the Codex prompt is insufficient for an edge case or the user explicitly asks to align with the Copilot wording.

## Database and Prisma (Migration-First)

- Never manually edit `apps/api/prisma/schema.prisma`.
- Never run `hedhog dev apply` in this `hub` repository.
- Never run commands that reset/recreate the project, database, schema, or existing migrations.
- Do not delete, overwrite, or regenerate existing migrations; they must remain intact for production database updates.
- For any database structure or seed-data change in `libraries/*/hedhog/table/*.yaml` or `libraries/*/hedhog/data/*.yaml`, create a new SQL migration under `apps/api/prisma/migrations` that mirrors the YAML change.
- After creating a new migration file, apply it by running `pnpm prisma:deploy` from `apps/api`.
- Refresh Prisma artifacts with `pnpm db:update` only when applicable after the database has the expected structure.
- Do not use `prisma migrate dev` as the repository workflow unless a future explicit project policy replaces this one.
- When authoring or editing `hedhog/table/*.yaml` or `hedhog/data/*.yaml` files, use the `Hedhog YAML` agent for schema-correct, validated YAML output.

## Endpoint Permission Consistency

- Backend endpoint changes must keep route and role seeds synchronized.
- Do not leave partial changes between controllers, routes, roles, and frontend consumption when they are part of the same flow.

## New Library Bootstrap

- Create new libraries using `hedhog dev create-library --name <library>` instead of manual scaffolding.
- Supported options: `-n, --name <name>`, `-f, --force`, `-v, --verbose`, `-s, --skip-install`.
- After creation, always run `pnpm install` at the repository root.

## Admin Asset Sync (Mandatory)

- If changes affect files under `apps/admin/src/app/(app)/(libraries)` or `apps/admin/messages`, run `hedhog dev assets-to-library` before completion.
- Pass affected library names without duplicates, for example:
  - `hedhog dev assets-to-library core`
  - `hedhog dev assets-to-library core lms finance`
- Infer library names from path segments:
  - `apps/admin/src/app/(app)/(libraries)/<library>/...`
  - `apps/admin/messages/<library>/...`

# Admin App

These instructions apply to `apps/admin`.

## Page structure

- All pages must render `Page` containing `PageHeader`.
- Maintain consistency with the layout and components already existing in the app.
- For CRUDs/listings, follow the compact density and visual distribution pattern described below; use the first generated page under `apps/admin/src/app/(app)/(libraries)/<library>/<entity>/page.tsx` as a concrete reference once it exists (this folder is empty in a fresh checkout of the template — it is generated by `hedhog dev create-library` / `assets-to-library`).

## Listing pattern

- Use the first generated page under `apps/admin/src/app/(app)/(libraries)/<library>/<entity>/page.tsx` as the canonical CRUD/listing layout reference once it exists.
- For pages with search/filter + alternate visualizations, keep a single flat toolbar row: `SearchBar` on the left and the `Tabela/Cards` `ToggleGroup` on the right.
- Do not wrap the toolbar or the main list block in an extra parent `Card`, and do not add duplicate list titles/descriptions below `PageHeader` unless the flow truly needs a separate contextual alert.
- For listing pages, follow the visual flow: `PageHeader` -> `KpiCardsGrid` (when there are KPIs) -> `SearchBar` -> list -> `PaginationFooter`.
- On clickable rows, cards, or list items, apply `cursor-pointer` (or an equivalent style) as a mandatory visual indication of interactivity.
- The primary registration action must live in `PageHeader.actions`.
- In CRUD listings, the primary action should normally open a right-side `Sheet` for create/edit.
- When useful metrics exist, aim for at least 4 KPIs or more to keep the same visual density across screens.
- The `SearchBar` should sit right below the KPIs, with search, filters, and the search button aligned on the same row when space allows.
- The filter row may also include dates, groupings, and other recurring filters, but it should remain a single horizontal block when width allows.
- Do not place the primary registration button in the same horizontal block as the `SearchBar`; the row below the KPIs should be reserved for search, filters, and the `Search` button.
- Render the items right below, in a table or card grid as appropriate.
- When there are no items, render a standard empty state with a light gray dashed border, a centered icon, title/description, and a primary button to create the first record.
- Prefer a table for operational data with multiple columns, badges, and inline actions; use a grid/card layout for more visual items.
- In rows or cards, keep a recurring language: avatar or logo when applicable, primary name with a secondary gray subtitle such as email, compact badges, and icon or small-button actions.
- Avoid a main container `Card` wrapping the search/filters and the entire listing; prefer a flat structure like the users screen and the `catalog` listings.
- Reuse the components from `apps/admin/src/components/entity-list` first before creating another structural variation.

## Data and context

- Use `useQuery` and `useApp` from `@hed-hog/next-app-provider`.
- Do not import `@tanstack/react-query` directly.
- Use `request` from `useApp()` for authenticated calls to the backend.

## Forms

- All forms must use `react-hook-form` with `zodResolver`.
- All fields must be rendered with `FormField` and Shadcn form primitives.
- Avoid plain HTML forms or uncontrolled inputs.
- Reuse the same component for create and edit when doing so reduces duplication without harming clarity.
- In CRUD listings, prefer a side `Sheet` for create/edit instead of a dialog with raw JSON.
- The primary create/edit button must have contextual text per entity, such as `New Brand`, `New Site`, `New Product`.
- Fields must occupy the full available width of the block/column they are in.
- Add guiding placeholders on fields; on rich fields, expose an equivalent hint when a native placeholder is not available.
- For frequent relations, use an autocomplete/combobox with search, selection, clear selection, and, when it makes sense, inline creation in a new `Sheet`.
- For searchable entity relation fields with inline creation, use the shared `EntityPicker` from `@/components/ui/entity-picker` as the default implementation. Do not create new library-scoped `*FieldWithCreate` or `*SelectWithCreate` variants unless they are thin semantic adapters over `EntityPicker`.
- For admin forms that need drafts, use the shared hook at `apps/admin/src/hooks/use-form-draft.ts` instead of isolated local logic. Drafts must remain in localStorage with per-user isolation, restore only for the same user after involuntary session expiration, be cleared on successful submit, and removed on manual logout.
- Never persist passwords, tokens, or raw files/blobs in the draft.
- Reserve a plain `Select` for enums/status; do not use a plain select for relations if a shared autocomplete component already exists.
- For upload fields, include progress, visual feedback for the linked file, replace, and remove.
- For image upload, include a preview of the current image and a visual placeholder when empty.
- In the `Sheet` footer, prefer only the primary full-width submit button when closing via the `X` is sufficient.
- Editable description fields should prefer `RichTextEditor`.
- `Switch` must be rendered without an extra fake input frame.
- `RichTextEditor` must stay contained within the `Sheet`, respecting the container width and without horizontal overflow.
- For logo/avatar, prefer a small square preview next to the upload actions, without a large container card.

## Shared components

- Generic, domain-independent components reusable across multiple flows must live in `apps/admin/src/components`.
- Components specialized for a module can live in the module itself and still be imported by other modules when that reuse makes sense.
- If a component stops being specific to a module and starts serving the admin broadly, promote it to `apps/admin/src/components`.
- Before creating a new component, check whether the case can be handled by an existing shared component or by a small extension of a current component.

## Confirmations and UX

- For destructive or irreversible actions, use Shadcn's `AlertDialog`.
- Do not use `window.confirm`.
- Keep text localized with `useTranslations()`.

## Tables and listings

- In pagination footers, follow the responsive pattern already adopted in the project.
- Reuse existing components, hooks, and visual conventions before introducing a new variation.
- In reports and dashboards, use color palettes with distinct colors per series/category and avoid monochromatic bars/pies.

## Utilities

- For image by id, use `getPhotoUrl` when that is the standard at the point of use.
- To get the user's email, use `getUserEmail` when applicable.

## Assets for libraries

- Whenever there are changes in `apps/admin/src/app/(app)/(libraries)` or in `apps/admin/messages`, run `hedhog dev assets-to-library` with the affected libraries.
- Determine the library from the path:
  - `apps/admin/src/app/(app)/(libraries)/<library>/...`
  - `apps/admin/messages/<library>/...`
- Examples:
  - `hedhog dev assets-to-library core`
  - `hedhog dev assets-to-library core lms finance`
